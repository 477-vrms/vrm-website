System Level Diagram

As with any engineering undertaking, it's critical that everyone on the team is on the same page in terms of their understanding of our system's understanding. To aid with this, as well as at the request of course staff, I made a very simple system level diagram to aid in the visualization of our systems functionality. In particular, our system has multiple components that will be in constant communication with each other using a variety of protocols. Some of this communication will be direct, such as transmitting video feed from the robot arm to the VR headset, while others will require some computation, such as when converting hand remote position information from the VR headset to arm movement instructions that the robot arm can carry out. Our first draft of our system level diagram can be seen below.

This system diagram shows the general flow of data throughout the components of our system. As shown, the VR headset and controller will communicate with the arm through the assistance of a Google Cloud virtual machine and a Raspberry Pi. The Google Cloud VM will take positional data from the VR headset and convert it to movement instructions for the robot arm, while simultaneously streaming video from the arm to the VR headset. TCP Websockets will be used to transfer video streaming data, while TCP MQTT will be used to transmit movement data, as MQTT is better suited for efficient transfer of smaller data sizes. The Raspberry Pi will be used to interface the hardware on the arm with the wireless connection to the Google Cloud server. We have discussed the possibilities of using UDP for communications in the future, but we think that this will be a good launching point for us to begin with.

DevOps Continuous Integration and Deployment with API Gateway

Our project will rely heavily on an API for communications between components. This API can be found here . It is a based off of the ECESS (Electrical and Computer Engineering Student Society) API gateway developed by Matthew and me, which can be found here . A big reason that we copied a large portion this API over is because it handles a lot of the housekeeping that we will need to be dealing with constantly with our code, much of this having to do with the continuous Integration pipeline that we want to have set up for our project. It will also allow us to interface with Google Cloud Platform for the deployment of our server-side code, which will become the backbone of how data is handled in our project (as seen in our system diagram).

Packaging

No matter the hardware, in order to safely deliver the product, everything must be packaged in a way that will protect the internals of the product and assist in its usability. For our project, the main feature will be an exposed robot arm, which doesn't need any packaging. So, the packaging for our product is going to be fairly simple, as the only thing that really needs protection is our PCB, Raspberry Pi, and any other exposed hardware for the arm. To do this, we can use a simple box that will protect these elements while providing an opening for any cables or connectors. There are many of these currently on the market that we could simply buy instead of having to make our own, but to illustrate what we are looking for I used CAD to make an example to our dimensions. I used a parameterized version of https://github.com/477-vrms/vrms-api/runs/4872020682?check_suite_focus=true design to fit our needs of a 13x10x8cm box. We plan on mounting this to the base of our robot arm.

PCB Layout Footprint

While there is a lot to PCB design, I was tasked this week with creating a very simple layout footprint for our PCB. Nearly all of the design still has to be done, but this gives us a general idea of what we need to include on our PCB. For our PCB we're going to keep it simple, since we will have a lot of help from our Raspberry Pi. Our PCB will include our STM micro alongside a buzzer and its MOSFET, with 5 screw terminals (3 for the arm, 2 for the buzzer). The layout can be seen below. All of our elements will fit on a 5x8cm board.

PCB Updates

As you can see from the image above, our PCB looks quite a bit different from last time. Joe provided us with a number of really helpful changes this past week. For starters, I beefed up the power/ground traces in order to better support our current flow. Next, I changed the size of all of our capacitors and resistors to 0805. I also moved in the decoupling capacitors for more efficient power flow. Finally, I added a ground plane with a proper clearance and mounting holes with exclusion zones. There were also some changes on the schematic side that Emma took care of, but I still had to update the PCB to match the schematic. These changes include adding some pull up/pull down resistors, adding current limiting resistors on the UART connections, breaking out some spare GPIO pins, adding a ground signal to the UART lines, and updating the buzzer footprint to match the actual buzzer we will be using. The only thing left in the schematic is to implement the module, which I will then replicate on the PCB.
