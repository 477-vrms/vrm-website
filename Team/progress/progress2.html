Main Goals for the Week

My main goals for the week were to get more familiar with Unity, and to be able to create movable arms within Unity. Since the arm relies on making a simulation for its movements, this task was top priority to me.

Unity

To get started with Unity and VR development, I did a couple tutorials teaching me how to interface the headset and controllers within the workspace. This also showed me a variety of features unity provides with object creation and management.

Creating Robot Arms

After learning a bit about how VR development works within Unity, I started making a simple arm. For this arm, I followed some tutorials on Inverse Kinematics within Unity to figure out how to create this arm. I wrote two scripts, IKmanager.cs to calculate the overall Inverse Kinematics in 2D, and LookAt.cs, to track the position of the target and "look" at it, which allows the arm to track a target in 3D.

After doing that, I remade the arm to give it a more 3D look, as well as to sharpen my skills in creating arms.

These two arms seemed to work well, and will probably be used to test our base level arm since they have the same number of joints and rotation axis. However, one of my goals for the project was to create an arm simulation that allowed for more control, and these two didnt allow for much other than location tracking. What I wanted was rotational tracking in all axis as well. To do this, I decided to not spend time trying extremely complex inverse kinematics for multiple targets, so I instead looked to unity packages that can handle this part for me. Here's an example of one attempt that took way too long to get working.

Unfortunately, this arm would only work inside a simulation, since once of the joints is unable to move properly on one axis. My search continued, with lots of testing, and I eventually found a package that easily allowed me to create a simulation with Inverse Kinematics for multiple targets. This current arm is the goal for the project, if we can succeed at adding extra joints to our robot arm.

This arm feels very natural to use, so I think the only changes to this arm will be to make it look nicer. Mechanically, the arm simulation seems to be done.

Overall, I've learned a lot about Unity, and how to create software for VR devices. There are a lot of complications that I had to work out, but those occur less now that I have more experience.

My main goals for the week were to get the Unity simulation to fully be able to send all the data it needs to the server. This is important because the entire point of the simulation is to send the angular data to the server.

With help from Matthew, I managed to get a websocket set up in Unity, allowing us to send the data from the simulation to the server. It works well, but does have some little problems, like lag when too much data is sent. (Like if the arm moves too fast) These should be easily fixable, thankfully.

Getting robot claw to work

The next step was to figure out how to find how much the trigger is being pressed. Through a bit of research into unities libraries, I found a way to track it. I created a custom game object with a script to track the percent the trigger is pressed, and relayed that information to the connection script. So now Joint 8 is being broadcasted to the server.

After doing that, I made a custom model to get an animation of the claw. This claw has a script that reads from the trigger game object, and uses that percent pressed value to determine how much the claws need to move.

Now the robot arm actually feels like something that can pick up and move objects, and technically could if the physical arm was connected at this point.

Main Goals for the Week

My main goal for the week was to create a small demo for the arm within unity, while improving existing scripts.

Creating the demo

I created a small scene with some objects on a table that the arm is able to grab. Each object has physics, so the arm can interact with the scene just like a real life arm could interact with real life objects. While the physics are a little wonky, I think this is a good demo for showing how to use the simulation.

New Scripts

I created a bunch of new scripts to help with this demo, and some to improve the general use of the arm:

AllowClawToGrab.cs - This script is a big one, and required a lot of fine tuning and obscure knowledge of how Unity works. What this does is allows the claw to "grab" objects to showcase what the real arm should be able to do. The script checks certain collision polygons on the arm while its closing, and if those polygons intersect an object, it will stick the object to the claw, and stop the claw from closing any more. This gives the illusion of the claw grabbing the object. When the user lets go of the trigger, the claw opens, and drops the object. The object is also given velocities to reflect how the arm was moving the object. This script is a little glitchy since unity's physics system doesn't really like collisions between two physics objects very well, but it works well enough for a demo to showcase the simulation.

RemoteControl.cs - This script allows the user to remotely control the robot arm target. By holding a button on the controller, the user gets full control of the target, and can move it without bringing the controller all the way to the target.

Reset.cs - This script allows the user to reset all of the objects in the demo scene by pressing a button on the controller. This is helpful because when playing the demo, it is very easy for objects to become too far away from the arm.

LimitRotation.cs - This script limits the amount the target can be rotated, as an extra layer of protection for the joint limits.

General Ignore Collision Scripts - These scripts help stop things going crazy by disabling collisions with certain objects like the player object and some parts of the arm.

Modified Scripts

I also modified a few of my existing scripts to work better:

TriggerGrip.cs (Trigger button to control claw) - Changed the way the trigger controls the claw, added acceleration to the claw movements and gave it more precision.

Connection.cs (connection to server) - Reduced the number of bytes the JSON packet needs to send. Added an epoch timestamp for synchronization.

Week 5:

Date: 2/11/2022
Total hours: 15
Description of design efforts:

Synopsis

This week, I optimized some scripts, started on the debug menu, and changed the number of joints on the arm.

First, I did a bit of bug-fixing to make the simulation run faster and transmit data better. Previously for reasons unknown the firebase was updating slowly when transmitting data from the lab PC and standalone VR headset. What I figured out was that it was a mix of unoptimized scripts and Wi-Fi latency. I’ve properly optimized the scripts, and we bought a USB-C to ethernet adapter for the headset if Wi-Fi is ever a problem. For the debug menu, I started working on it. This took a lot of research into how Oculus does UI handling, so it was a lot of guessing and checking what worked and what didn’t. I ended up getting it working, and all I need to do now is add features to it. For the robot arm, I changed the simulation to only use 7 joints, which means that we only need to add one extra motor to the robot arm. I’ve found by testing that this joint was somewhat redundant, and doesn’t change the functionality that much.

Optimizing

This week, I worked on optimizing the general scripts I used, as it was clear that some of these had some less than ideal complexity levels. The main problem this was causing is that the standalone VR headset was transmitting data very slowly, which is against one of our PSSCs. I found that the main culprit was actually an optimization I made to limit the amount of data being sent to the server, by only sending it every hundredth of a millisecond, as opposed to as fast as possible. This ended up lagging the program due to the calculations I was doing. I removed this, and added a different optimization and now it works as well as the more powerful computer I was testing with.

Debug Menu

I started work on the debug menu, which took a much longer time than expected. This required a lot of tutorials, and forum posts to get working. Eventually, I figured it out. While this is currently just a debug menu to toggle features, I think that this is the start of a main menu, and various UI elements the simulation definitely needs.

Robot Arm

Previously, we thought that the robot would need two extra joints to work well, bringing it to 8 total joints. I found that the 4th joint is somewhat redundant, since joint 6 does most of the same work. I locked the joint to 0 rotation, and it looks like I was correct. Going forward, we are going to be only using 7 joints, since it will be much easier to make mechanically.

What I Learned

I learned this week that I have to be smart about what I'm writing in my scripts, since I see now that they can cause performance issues. I also learned how UI works with oculus controllers, which will be important for later.

Next Week

Next week, I plan to work on the debug menu further, and more UI elements in general. Maybe if the Pi camera gets sent to the server I can work on that, but I currently don’t know the ETA on that.

Menu

I added a bunch of features to the menu, along with making sure that they actually worked. This required changing scripts and adding a workaround to make everything work as needed.

Here is a rundown of what each thing does:

Box Stacking Demo - Toggles all of the objects from the "box stacking demo" which is just a huge arm with some boxes you can stack on top of eachother.

Table Demo - Toggles the original demo, the table with a couple objects that you can interact with.

Joint Text Data - Toggles the list of rotational angles of the joints.

Main Arm Toggle - Turns the arm on/off.

Server Connection - Toggles the connection to the server.

Controls Menu - Toggles the controls menu.

Main arm RC Multiplier - This slider controls how fast you want the remote control to move the target. Higher values means that you have to move your arm less to make the target move.

Main Arm RC Rotation Lock - This is a bit confusing, but it snaps the rotation of the target to the controller. With it off, the arm will not snap, and will allow you to control the rotation of the arm without it snapping to your rotation. (It can get very hard to control with it off)

Instant Claw - If on, will make the claw move instantly with the trigger, instead of an acceleration effect.

Test Arm - Makes the arm move by itself, somewhat like a demo. This can be used to show the arm moving without having to control it.
